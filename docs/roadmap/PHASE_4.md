## Goal

Build the L-side (cost) of P&L on existing studio entities and finish studio operational autonomy so studios stop depending on `/system/*` routes. Phase 4 produces **read-only reference figures** — no money moves, no payment processing, no bank reconciliation, no dispute workflow.

This is the approval path that actually **populates** `Show` / `ShowCreator` / `ShowPlatform` actuals from bound task fields — the review surface below only reads them.
- ✅ **PR 12.4.4 — Operational facts summary and filters** ([#111](https://github.com/allenlin90/eridu-services/pull/111)): Show Run Review (`/show-run-review`) summarizes confirmed extracted facts for an operational day/range — show-actuals completeness, late/missing creators with reasons, active platform violations, and incomplete phase checks — across four URL-synced DataTable tabs (creators / violations / tasks / shows) with per-tab search and status filters. Operational-day window math moved fully to the frontend: the client computes 06:00→05:59 local bounds via shared `operational-day-range` utilities and serializes absolute ISO-8601 strings; the backend `GET /studios/:studioId/shows/run-review` endpoint stays timezone-agnostic, validates the explicit bounds, and caps the window at 31 days to bound the in-memory aggregation (FE surfaces the returned validation message). **Read-only**: it reports the state of already-extracted `Show` / `ShowCreator` / `ShowPlatform` / `ShowPlatformViolation` columns and never writes them; actuals are populated upstream by the extraction pipeline (12.0.5 / 12.1.1) on task approval (12.4.3). Show-actuals reporting is **start-based**: a show counts as started once `actualStartTime` is recorded (the primary lateness signal), and the card surfaces late-start count plus total missing duration (`actualStartTime − planned startTime`); end-time recording is shown as secondary detail rather than gating completeness, since `show_actual_start_time` and `show_actual_end_time` are independent fact-key bindings.
  - **Follow-up → 12.4.6**: review surfaced that bulk approval silently extracts nothing when a completed task's frozen snapshot has no fact-key binding (e.g. generated before the binding was added) or empty bound fields, leaving actuals empty with no explanation. Surfacing "0 facts will be extracted" / flagging snapshot binding-drift at `/task-review` is folded into 12.4.6's stale-input resolution scope (deliverable added to [PRD §C 12.4.6](../features/task-fact-binding.md#section-c-review-surface-pr-124)).
- ✅ **PR 12.4.7 — Show Run Review CSV export** ([#120](https://github.com/allenlin90/eridu-services/pull/120)): each `/show-run-review` tab (creators / violations / tasks / shows) gets an Export CSV action that refetches the tab's full filtered set (`limit = total`, same active `search`/`status`/`severity`/`completeness` filters) and serializes **every matching row — the full filtered result, not the current page**. Client-side serialization via the shared `@/lib/csv` + `@/lib/file-download` utils; no new endpoint. Per-tab pending/disabled state; no-op when the filtered total is 0.
- ✅ **Perf pass — task-setup / task-review / show-run-review** ([#119](https://github.com/allenlin90/eridu-services/pull/119)): not a roadmap deliverable — a cross-cutting optimization of the 12.4.x surfaces. `/task-review` tab badges move from client-side aggregation of a fetch-all dataset to a single `GET /tasks/review-stats` count endpoint, and the table tabs become server-filtered via `review_tab`; the stats query scopes by "due-date-in-range OR undated-with-show-in-range" so undated review tasks are still counted. Show Run Review splits its monolithic nested response into a lean summary plus four lazy paginated sub-resources (`run-review/{creators,violations,tasks,shows}`), with summary counts and list rows derived from shared helpers so they can't drift. Task lists drop the heavy JSONB `snapshot.schema` from the studio review list (lazy-loaded via the task detail endpoint) while the member `/me/tasks` execution path keeps it (`taskListIncludeWithSchema`). Pattern + guard rails documented in [`.agent/skills/api-performance-optimization/SKILL.md` §8](../../.agent/skills/api-performance-optimization/SKILL.md).
- ✅ **PR 13 — Studio-shifts export loading-state race fix** ([#124](https://github.com/allenlin90/eridu-services/pull/124)): moved `setIsExporting(false)` inside the `exportAbortRef.current === controller` guard so an aborted, superseded export can no longer clear the loading flag mid-flight; mirrors PR 4's `/task-setup` fix and the `useAbortableAction` hook.

### PR 20 · Direction (2026-06-07 reset)

The original 20.1–20.5 plan (heavyweight client catalog with immutable versioning + campaign sets, derived from [#87](https://github.com/allenlin90/eridu-services/pull/87)) was retargeted to a **lean catalog + reviewer** shape after a design pass. Locked decisions:

- **Data model: keep the client-owned catalog (drop PR#86's template-local storage), but ship lean** — a single mutable `ClientMechanic` per client. **Immutable `ClientMechanicVersion`, `ClientMechanicCampaignSet` + items, cross-studio copies, and per-template label overrides are deferred** (see [Out of scope](#out-of-scope-post-phase-4)), to be added when client-facing access / multi-campaign curation becomes concrete.
- **Catalog scope (B2): client-global, studio-authorized writes.** `ClientMechanic` is owned by the global `Client` (single truth across studios); write access is granted only to studio members linked to that client. Cross-studio propagation is intended, not a bug.
- **Template binding (B1): mechanic-bearing templates are client-bound.** `TaskTemplate` is studio-scoped today; a template that uses mechanics gains an optional `clientId` so one client's cues never leak onto another client's shows.
- **Role (B3): `ACCOUNT_MANAGER` edits the catalog, read-only on operations.** Writes mechanics/catalog; read-only on task templates, shows, and creator mapping with money fields redacted; no access to members / shifts / compensation / economics. ADMIN/MANAGER retain catalog write.
- **Coverage is the account-manager's core job and is bidirectional**: mechanic→shows ("is this mechanic on the latest template and reaching my target shows?") and show→mechanics ("is this show running the right, current mechanics?").
- **Staleness without versioning (S1):** a monotonic `contentRevision` int on `ClientMechanic` (bumped on content edit) is frozen into each template snapshot's `mechanic_ref`, so coverage compares frozen-vs-current revision exactly — no version history needed.
- **Coverage is queryable (S2):** a denormalized `TaskTemplateMechanicRef` link table (template/snapshot ↔ mechanic + loop), written on template save, backs both coverage directions — never a JSONB scan.
- **Money redaction (S3): allow-list projection, not field blacklist**, with a negative golden test enumerating money-bearing fields (Finance Guardrails).
- **Duplicates / lifecycle:** per-loop `(mechanic_id, group)` uniqueness (assign-once per loop; re-assign is a no-op); genuine duplicates live as distinct catalog entries or manual fields. Mechanics retire (soft); hard-delete is blocked while referenced (S4).

#### PR 21.8 · Performance trend operational-day bucketing fix

**Brief** — Surfaced during PR 21.6 review. `StudioPerformanceService.getPerformanceSummary` builds the daily trend by slicing the UTC calendar date off each timestamp — both the bucket keys (`curr.toISOString().slice(0, 10)` while walking `start_date → end_date`) and the per-show assignment (`show.startTime.toISOString().slice(0, 10)`). The frontend sends **absolute ISO bounds** derived from the local 06:00–05:59 operational day (PR 12.4.4 pattern), so for any non-UTC studio the trend's day boundaries fall at UTC midnight rather than the operational-day boundary: the first/last buckets are off by one and shows near local midnight land in the wrong day.

**Fix** — Bucket by an explicit, timezone-aware operational-day key instead of a raw UTC `.slice(0, 10)`. The backend stays timezone-agnostic for validation, but day-grouping must use the same operational-day definition the frontend selected (pass the studio/operational timezone or the precomputed day boundaries through to the aggregation), not the server's incidental UTC calendar. `slice(0, 10)` on an ISO string is **not** a date-bucketing primitive — it silently assumes UTC. Guard rail captured in the [`operations-review-surface` skill](../../.agent/skills/operations-review-surface/SKILL.md#operational-day-bucketing-never-slice-utc-iso) and project memory.

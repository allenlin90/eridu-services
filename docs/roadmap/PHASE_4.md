### PR 12 · Critical task-input semantics for actuals and performance

**Brief** — Task templates can already collect operational facts, but those values are only identifiable by snapshot schema and `task.content` keys. The system needs a template-level binding contract before it can safely treat operator-entered task fields as show actuals, creator attendance, creator actuals, platform actuals, violation records, or performance metrics. Add a shared schema attribute such as `critical_input_key` / `system_fact_key` with scoped enum values, builder/review UI for those bindings, validation that field type and target scope match the selected fact, and an engine-aware extractor that reads from the task snapshot rather than from template-local labels.

Task submission inputs remain part of the workflow even after extraction writes selected values into `Show`, `ShowCreator`, `ShowPlatform`, or `ShowPlatformViolation`. The task submission is an input record and audit/reference source; the scoped model field or child record is the current operational source of truth. For comparable facts, the selected model value follows source priority: platform data or platform upload > creator input > operator task input > planned schedule. When a higher-priority source arrives later, such as seller-center API integration or platform data upload, it can replace the selected model value while preserving the lower-priority task submission as reference.

Initial fact keys should cover:

| Fact key | Expected field type | Target scope | Notes |
| --- | --- | --- | --- |
| `show_actual_start_time`, `show_actual_end_time` | `datetime` | `Show` | Used only when the task records one overall event timeline. |
| `show_creator_actual_start_time`, `show_creator_actual_end_time` | `datetime` | `ShowCreator` | Captures creator-specific attendance timing and separates creator facts from show/platform facts. |
| `creator_attendance_status` | `select` | `ShowCreator` | Values start with `on_time`, `late`, `missing`; reason sidecar is required for non-`on_time` values. |
| `show_platform_actual_start_time`, `show_platform_actual_end_time` | `datetime` | `ShowPlatform` | Captures seller-center/platform stream actuals when the platform is identifiable. |
| `platform_gmv`, `platform_view_count` | `number` | `ShowPlatform` | First typed platform performance columns. Additional metrics such as CTR/CTO stay report-only until a later migration promotes them. |
| `platform_violation_observed` plus violation detail fields | `select` plus text/enums as designed | `ShowPlatformViolation` child rows | `have_violation` creates one or more platform-scoped violation records. `no_violation` is a review signal and does not create a violation row. |

Do not infer facts from labels such as `MC attended`, `Violation_status`, `GMV`, or `Start Time`. Labels and descriptions are user-facing copy; system extraction must depend on explicit bindings frozen into the task-template snapshot. The design doc should land before PR 12.1-12.4 and should update the task-template feature-doc maintenance set if the shared schema changes. It must also define the input-record shape, selected-source metadata, and overwrite/reconciliation policy for later higher-priority sources.

### PR 12.1 · Operator-recorded show and platform actuals via CLOSURE task template

**Brief** — PR 4 made show `actual_start_time` / `actual_end_time` editable on `/show-operations`, but that surface is for admin/manager audit. The operational input sources are the operator running the show, creator-attributed inputs, and the platform seller center/API. Add CLOSURE task-template fields bound to `show_actual_*` when the task records one event timeline, and `show_platform_actual_*` when the task records platform-specific stream timing. Extract these fields immediately on task submission, validate the same no-inverted-range invariant as `/show-operations`, and write the resulting timestamps to `Show` or `ShowPlatform` according to the binding and source-priority policy.

Admin/manager `/show-operations` remains the audit/override surface. The implementation must define how direct show actuals and platform actuals roll up in read models, how manager edits and operator submissions reconcile, how `/show-operations` displays task-sourced actuals, and what happens to already-closed shows that only have legacy task content. Operator task inputs must still be retained after extraction; they become reference inputs when a higher-priority platform or creator source later selects a different model value.

### PR 12.2 · Creator actuals and attendance extraction from task submissions

**Brief** — `On air_check`-style ACTIVE tasks already collect `MC attended` values such as `on_time`, `late`, and `missing`, with a required reason for non-`on_time` values. Promote creator attendance and actual start/end values from anonymous task content into system-readable `ShowCreator` facts. The task schema must identify the target creator assignment, either through per-creator fields, a row/repeater-style binding, or a manager resolution queue when legacy task content only has one show-level MC field.

Creator facts stay separate from `Show` and `ShowPlatform`: a creator can be late, missing, or have a different participation window even when the platform stream and overall show ran normally. Legacy single-select attendance can only be auto-attached when the show has exactly one active `ShowCreator`; otherwise it must remain unresolved until a manager maps it. Creator-entered or creator-attributed inputs outrank operator task inputs for comparable creator facts, but platform data still outranks both when the platform can provide the same actual/performance fact.

### PR 12.3 · Platform GMV/views and violation extraction

**Brief** — `Post_production_check`-style CLOSURE tasks already collect `Violation_status`, `GMV`, `View`, `CTR`, and `CTO`. Promote GMV and views into typed `ShowPlatform` columns because those are the first performance facts with clear storage ownership. Keep CTR/CTO in task reports for now; they can be promoted by a later migration when the reporting and product semantics are stable.

Violations are not a single status column. Add a `ShowPlatformViolation`-style child model so a `ShowPlatform` can have zero, one, or many violation records. Extraction must be snapshot-driven and idempotent: re-submitting a task updates the scoped platform GMV/views and upserts or supersedes violation rows according to a documented policy, while historical task content remains the audit source. If the task cannot resolve a single `ShowPlatform`, the record should enter the review queue rather than being written to `Show` as an approximation. The report path can continue reading task submissions, but operational performance surfaces should read the typed facts once this lands.

### PR 12.4 · Actuals and performance review sign-off

**Brief** — Task submissions write scoped facts immediately; manager approval is a review workflow over persisted records. Add a date-range summary/table view where managers can review daily, weekly, or custom-range actuals and performance facts, with abnormal rows highlighted for attention. The view should support bulk sign-off for the selected range rather than requiring one task-by-task approval flow.

Abnormal highlights should include at least missing or incomplete actual pairs, creator `late` / `missing`, new or unresolved platform violation records, GMV/views with zero or extreme values, unresolved creator/platform mappings, and conflicts between direct show actuals and platform-derived actuals. The sign-off record should describe the range, actor, timestamp, filters, and unresolved abnormal count so the operation has an audit trail without blocking initial writes.

### PR 17 · Entity edit dialogs → dedicated routes (audit + roll-out)

Each conversion gets its own scoped PR (one route per PR) for review tractability; this row covers (a) the audit of every conversion candidate with the target route shape and (b) the first conversion (recommend creator roster as the pilot since the compensation review dialog is already a heavy modal that wants tabs).

### PR 18 · Creator-rate wire-type migration

**Brief** — Surfaced during PR 5 review (Codex P2 on `updateStudioShowCreatorInputSchema`). `defaultRateInputSchema` and `defaultCommissionRateInputSchema` in `packages/api-types/src/studio-creators/schemas.ts` are both `z.coerce.number().…` (the canonical creator-rate wire format), so any decimal string sent from the FE is coerced through a JS `number` before the controller converts it back to a `Prisma.Decimal` string at the persistence boundary. For realistic rate values this is exact (everything is many orders of magnitude below `Number.MAX_SAFE_INTEGER`), but it's the last creator-side money path going through JS number and is inconsistent with the `Big`/string contract finalized in PR 1 + Finance Guardrail #2. Migrate both helper schemas to `z.string()` end-to-end across all three consumers (`createStudioCreatorRosterInputSchema`, `updateStudioCreatorRosterInputSchema`, `updateStudioShowCreatorInputSchema`), keep the `superRefine` cross-field invariants, update the controllers/services to parse decimals at the boundary (drop `toMoneyString` round-trip on the way out), and update the studios FE forms / payload builders (`buildShowCreatorAssignmentTermsPayload`, `buildUpdateStudioCreatorRosterPayload`) to send strings directly. Pair with PR 8 for the member-side equivalent — same shape of change, same rationale. No DB migration required (creator rate columns are already `Decimal` in Prisma).

### PR 19 · System creator rate wire-type migration

**Brief** — Surfaced during PR 8 audit (2026-05-18). `packages/api-types/src/creators/schemas.ts` exposes the system-admin creator CRUD. The response schema (`creatorApiResponseSchema`) already serializes `default_rate` and `default_commission_rate` as `z.string().nullable()`, but the input schemas (`createCreatorInputSchema`, `updateCreatorInputSchema`) coerce them through `z.coerce.number()` — so an admin POST/PATCH still round-trips money through a JS number even though the GET is string-safe. Migrate both input schemas to `z.string()` end-to-end, update the BE service/controller for `/admin/creators/*` (use `decimalToString` on the way out, parse decimals at the persistence boundary), and update any FE admin form that posts to these endpoints. No DB migration required (creator rate columns are already `Decimal` in Prisma). Independent of PR 18 (PR 18 covers the studio-scoped roster helpers); ship in any order.

### PR 20 · Show inline creator-rate wire-type migration

**Brief** — Surfaced during PR 8 audit (2026-05-18). `updateShowInputSchema` in `packages/api-types/src/shows/schemas.ts` accepts an inline `creators[]` array with `agreed_rate: z.coerce.number().positive()` and `commission_rate: z.coerce.number().min(0).max(100)` — these duplicate the `defaultRateInputSchema` / `defaultCommissionRateInputSchema` helpers that PR 18 migrates. After PR 18 lands, either (a) inline the new `z.string()` shape here, or (b) extract a single shared helper from `studio-creators/schemas.ts` and consume it from both places. Mirror the BE/FE updates done in PR 18. Depends on PR 18 so the helper shape is settled before this PR opens.

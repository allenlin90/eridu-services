### PR 12 · Critical task-input semantics for actuals and performance

**Brief** — Task templates can already collect operational facts, but those values are only identifiable by snapshot schema and `task.content` keys. The system needs a template-level binding contract before it can safely treat operator-entered task fields as show actuals, creator attendance, creator actuals, platform actuals, violation records, or performance metrics. Add a shared schema attribute `system_fact_key` with closed-system enum values, builder/review UI for those bindings, validation that field type and target scope match the selected fact, and an engine-aware extractor that reads from the task snapshot rather than from template-local labels.

This row is a **meta-row**: the design is locked here, but the actual implementation ships as four foundation PRs (12.0.1-12.0.4) plus the extractor PRs (12.1.1 / 12.1.2 / 12.2 / 12.3.1 / 12.3.2 / 12.4). See "Proposed sub-split for PR 12 (foundation)" below for scope and dependencies of each sub-PR. Status flips to ✅ when all sub-PRs merge.

**Design deliverable** — The finalized design has been locked and documented in [TASK_INPUT_FACT_BINDING_DESIGN.md](../../apps/erify_api/docs/design/TASK_INPUT_FACT_BINDING_DESIGN.md). All sub-PRs inherit these architectural decisions.

#### Design Decisions & Resolution Path

1. **Lean Core Models with Per-Field Source Map**: Core models (`Show`, `ShowCreator`, `ShowPlatform`) will **not** carry `actualsStatus` or row-level `actualsSource` columns. Instead, source tracking is stored as a **per-field map** inside the model's existing `metadata` bucket: `metadata.actuals_source = { actual_start_time: 'MANAGER', actual_end_time: 'OPERATOR', gmv: 'PLATFORM', viewer_count: 'PLATFORM' }`. This lets a row mix sources cleanly and resolves the `ShowPlatform.viewerCount` default-zero ambiguity (absent key = never written; present = submitted, even if zero). Actuals state is inferred dynamically from pair completeness: complete means both `actualStartTime` and `actualEndTime` are present; missing/incomplete means either edge is absent.
2. **Derived Attendance State**: Creator attendance state (`ON_TIME` / `LATE` / `MISSING`) is **derived at read time**, never stored as a column. `ShowCreator` stores `actualStartTime`, `actualEndTime`, a sticky boolean `attendanceMissing` (set explicitly by operator/manager when a creator did not show up), and `attendanceReason`. Lateness derives from `ShowCreator.actualStartTime > Show.startTime`; `MISSING` derives from `attendanceMissing = true`. Phase 4 has no grace window — 1 second over is `LATE`. This avoids dual-input conflicts between explicit status and explicit timestamps in the same operator submission.
3. **Standardized Polymorphic Auditing**: Centralize audits using a standardized polymorphic audit system. The new `Audit` model logs actions and changesets, while `AuditTarget` defines explicit, optional foreign key relations (`showId`, `showCreatorId`, `showPlatformId`, `studioShiftId`) to maintain strict database referential integrity. Manager overrides carry HTTP actor/IP/UA; extraction-engine writes carry `actorId = null` with `metadata.ingestion_source = "task_submission"`, `metadata.task_uid`, `metadata.task_field_id`. The legacy `metadata.audit.snapshot_overrides[]` pattern remains read-compatible via a sidecar reader; a backfill is tracked separately.
4. **Additive Snapshot Hydration**: Task templates remain generic. At task instantiation, a `TaskTarget` with `targetType: "SHOW"` links the generic task to the `Show` record. The generation engine inspects template fields with a `system_fact_key` and hydrates the task's snapshot schema with target-keyed input fields (e.g., `fld_attendance_missing_creator_<creatorUid>`, `fld_gmv_platform_<platformUid>`). Field keys are deterministic and stable — the same target always lands on the same key across re-hydrations. The snapshot is **append-only mutable** for hydrated bindings: re-rendering a not-yet-submitted task appends new fields for newly assigned targets, preserves existing fields whose targets are still assigned, and marks fields whose targets were unassigned afterwards as `binding_stale: true`. Stale-bound fields are skipped by the extractor and routed to the review queue (PR 12.4) instead of writing to an unassigned target. Non-hydrated template fields stay immutable.
5. **Event-Driven Push Model (Per-Field Priority)**: Facts are processed immediately upon a source input event (synchronous-on-submit for task submission, or manual manager override). Priority is evaluated **per fact**, not per row. Phase 4 active hierarchy:
   $$\text{MANAGER Override} > \text{PLATFORM / System Input} > \text{OPERATOR Task Input} > \text{PLANNED Schedule}$$
   A reserved `CREATOR_INPUT` tier sits between PLATFORM and OPERATOR for forward compatibility; it has no Phase 4 writer. The extractor compares the incoming source priority for each fact key against `metadata.actuals_source.<fact_key>`. If higher or equal, the target column is updated and the per-field source map is updated; if lower, the active value is unchanged but the lower-priority input is preserved in `task.content` and one `Audit` row with `action="SKIPPED_LOWER_PRIORITY"` is written so the review surface can explain non-writes. Manager overrides always win because system inputs can be wrong and compensation-impacting corrections need explicit human approval and audit.
6. **Closed Enum Extensibility**: The `system_fact_key` enum is closed-system (Phase 4 set only). High-frequency query items (actual times, violations, GMV, viewer count) are first-class indexed columns; all other metrics (CTR, CTO, concurrents) live in a flexible `performance_metrics` JSONB bucket on the relevant platform/show models. A migration cutover skill is established to promote JSONB metrics to indexed columns if required. `ShowCreator.performanceMetrics` is intentionally NOT added in PR 12 — it ships when a first writer exists.
7. **Read-Model Continuity**: Once typed model facts exist, task-report system columns (PR 4's `actual_start_time` / `actual_end_time` / `actuals_status`, plus any creator/platform analogs) read directly from the typed model columns, securing a single source of truth.
8. **Wire-Label Rename**: `OPERATOR_RECORD` → `OPERATOR_INPUT` in calculator output and FE labels. This is a public contract change to surfaces already shipped in PR 4 / 10 / 11; it lands atomically inside the PR 12 series and not as a follow-up.
9. **Legacy Show Backfill**: Policies adopt an opportunistic migration strategy: legacy shows stay as-is but are automatically backfilled when an old task is re-submitted or when a manager opens/edits the show, keeping data migration risk low.

#### Inline Decisions & Out of Design

- **Extraction Trigger**: Synchronous-on-submit to ensure validation errors are immediately surfaced to operators.
- **Re-submission Idempotency**: Upserts on actuals/performance facts according to per-field priority; **replace-all** for `ShowPlatformViolation` rows scoped to `(sourceTaskId, sourceFieldId)`. Manager-entered violations or violations from other tasks are never auto-superseded by a re-submission. Superseded rows are soft-kept with `supersededAt` for history.
- **Sign-off Granularity (PR 12.4)**: Single combined batch sign-off across a date range. Sign-off does not gate writes — it records that a manager reviewed the resulting facts.
- **Audit engine semantics**: Manager edits carry HTTP actor/IP/UA. Extraction-engine writes use `actorId = null`, `metadata.ingestion_source = "task_submission"`, `metadata.task_uid`, `metadata.task_field_id`. Lower-priority skipped writes emit `action = "SKIPPED_LOWER_PRIORITY"` audit rows so the review surface can explain non-writes.
- **Wire-label rename**: `OPERATOR_RECORD` → `OPERATOR_INPUT` ships atomically inside the PR 12 series (foundation PR), updating PR 4 / 10 / 11 self-views in the same change.
- **Currency on `ShowPlatform.gmv`**: `Decimal(12, 2)` is sized for THB. If multi-currency support is required before commission resolution lands (Phase 5), widen the column or store currency-scaled values as part of that workstream.
- **Out of Design (Deferred to Phase 5+)**: Direct creator-app telemetry, automated seller-center integrations, non-standard metric typed columns (CTR/CTO stay in JSONB), automated edit notifications, `ShowCreator.performanceMetrics` JSONB column (no Phase 4 writer), and commission payment engines.

### PR 12.1 · Operator-recorded show and platform actuals via CLOSURE task template

**Brief** — PR 4 made show `actual_start_time` / `actual_end_time` editable on `/show-operations`, but that surface is for admin/manager audit. The operational input sources are the operator running the show, creator-attributed inputs, and the platform seller center/API. Add CLOSURE task-template fields bound to `show_actual_*` when the task records one event timeline, and `show_platform_actual_*` when the task records platform-specific stream timing. Extract these fields immediately on task submission, validate the same no-inverted-range invariant as `/show-operations`, and write the resulting timestamps to `Show` or `ShowPlatform` according to the binding and source-priority policy.

Admin/manager `/show-operations` remains the audit/override surface. The implementation must define how direct show actuals and platform actuals roll up in read models, how manager edits and operator submissions reconcile, how `/show-operations` displays task-sourced actuals, and what happens to already-closed shows that only have legacy task content. Operator task inputs must still be retained after extraction; they become reference inputs when a higher-priority platform or creator source later selects a different model value.

### PR 12.2 · Creator actuals and attendance extraction from task submissions

**Brief** — `On air_check`-style ACTIVE tasks already collect `MC attended` values such as `on_time`, `late`, and `missing`, with a required reason for non-`on_time` values. Promote creator attendance and actual start/end values from anonymous task content into system-readable `ShowCreator` facts. The task schema identifies the target creator assignment via additive snapshot hydration: one set of fields per assigned `ShowCreator`, keyed by `ShowCreator.uid` (e.g., `fld_actual_start_creator_<creatorUid>`, `fld_attendance_missing_creator_<creatorUid>`, `fld_attendance_reason_creator_<creatorUid>`). When legacy task content only has one show-level MC field and the show has multiple active `ShowCreator`s, the field is routed to the manager resolution queue (PR 12.4) rather than auto-attached.

Attendance state is **derived**, not stored. The schema additions on `ShowCreator` are: `actualStartTime`, `actualEndTime`, `attendanceMissing` (sticky boolean), `attendanceReason`. `ON_TIME` / `LATE` derive from `ShowCreator.actualStartTime` vs `Show.startTime`; `MISSING` derives from `attendanceMissing = true`. Phase 4 has no grace window (1 second late = `LATE`). A reason is required when `attendanceMissing = true` or when the derived status is `LATE`; the extractor enforces this against the sidecar reason field before writing.

Creator facts stay separate from `Show` and `ShowPlatform`: a creator can be late, missing, or have a different participation window even when the platform stream and overall show ran normally. A show can start first while waiting for creators to join, so creator lateness is measured against scheduled `Show.startTime`, not inferred from `Show.actualStartTime`.

Phase-4 active source priority for creator facts is `manager override > operator task input (targeting a specific creator via the binding) > planned schedule`. A reserved `CREATOR_INPUT` tier sits between platform/system and operator inputs for forward compatibility but has no Phase 4 writer; the resolver enum reserves the value and the extractor does not emit it. "Operator task input targeting a creator" is not the same as a creator-sourced input; the binding model decides which creator the input is *about*, not who submitted it.

### PR 12.3 · Platform GMV/views and violation extraction

**Brief** — `Post_production_check`-style CLOSURE tasks already collect `Violation_status`, `GMV`, `View`, `CTR`, and `CTO`. Promote GMV and views into typed `ShowPlatform` columns because those are the first performance facts with clear storage ownership. Keep CTR/CTO in `ShowPlatform.performanceMetrics` JSONB for now; they can be promoted by a later migration cutover when the reporting and product semantics are stable.

Violations are not a single status column. Add a `ShowPlatformViolation` child model so a `ShowPlatform` can have zero, one, or many violation records. The model carries `sourceTaskId` and `sourceFieldId` so the extractor can scope re-submission behavior. **Re-submission policy**: replace-all scoped to `(sourceTaskId, sourceFieldId)`. A second submission of the same CLOSURE task supersedes only the violations it itself previously created (mark prior rows `supersededAt = now()` and write new active rows); manager-entered violations or violations from other tasks on the same `ShowPlatform` are never auto-superseded. One `Audit` row per supersession batch records the prior violation UIDs.

GMV and view-count writes follow the per-field priority resolver: each `metadata.actuals_source` entry on `ShowPlatform` (`gmv`, `viewer_count`) is checked independently, so a manager-overridden GMV does not block a platform-sourced viewer-count update or vice versa. `ShowPlatform.viewerCount` already exists with `Int @default(0)`; the per-field source map is what distinguishes a default zero from a submitted zero. If the task cannot resolve a single `ShowPlatform` from the hydrated field key, the value enters the review queue (PR 12.4) rather than being written to `Show` as an approximation. The report path can continue reading task submissions, but operational performance surfaces read the typed facts once this lands.

### PR 12.4 · Actuals and performance review sign-off

**Brief** — Task submissions write scoped facts immediately; manager approval is a review workflow over persisted records. Add a date-range summary/table view where managers can review daily, weekly, or custom-range actuals and performance facts, with abnormal rows highlighted for attention. This review surface is the Phase 4 home for filtering shows and creators by performance metrics and operational abnormalities; the filters are not part of planning. The view should support bulk sign-off for the selected range rather than requiring one task-by-task approval flow.

Abnormal highlights should include at least missing or incomplete actual pairs, creator `late` / `missing`, new or unresolved platform violation records, GMV/views with zero or extreme values, unresolved creator/platform mappings, and conflicts between direct show actuals and platform-derived actuals. The sign-off record should describe the range, actor, timestamp, filters, and unresolved abnormal count so the operation has an audit trail without blocking initial writes.

### PR 18 · Creator-rate wire-type migration

**Brief** — Surfaced during PR 5 review (Codex P2 on `updateStudioShowCreatorInputSchema`). `defaultRateInputSchema` and `defaultCommissionRateInputSchema` in `packages/api-types/src/studio-creators/schemas.ts` are both `z.coerce.number().…` (the canonical creator-rate wire format), so any decimal string sent from the FE is coerced through a JS `number` before the controller converts it back to a `Prisma.Decimal` string at the persistence boundary. Migrate both helper schemas to `z.string()` end-to-end across all three consumers (`createStudioCreatorRosterInputSchema`, `updateStudioCreatorRosterInputSchema`, `updateStudioShowCreatorInputSchema`), keep the `superRefine` cross-field invariants, update the controllers/services to parse decimals at the boundary (drop `toMoneyString` round-trip on the way out), and update the studios FE forms / payload builders (`buildShowCreatorAssignmentTermsPayload`, `buildUpdateStudioCreatorRosterPayload`) to send strings directly.

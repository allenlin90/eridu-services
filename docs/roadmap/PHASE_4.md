### PR 11.7 ¬∑ Task template loop **Grid view** for repeated mechanic cue cards

**Brief** ‚Äî Surfaced 2026-05-19 against template `ttpl_pWi1mbHEtHU0D-Zc3cHa` in studio `std_OBXMKm0gW4IGQUNQzp4E`. In loop-based moderation templates, each loop hosts multiple checkbox cue-card fields whose **labels** are the moderator's read-aloud copy (product name, promo mechanic). Authoring is painful because each loop card is rendered separately ‚Äî producers planning loops in Google Sheets must hand-transcribe each cell into a per-loop card, label drift between identical-looking cues goes unnoticed, and there is no way to fill-down or paste a TSV block. Add a per-template `Cards` ‚Üî `Grid` view toggle next to the existing **Workflow View** select in [`apps/erify_studios/src/components/task-templates/builder/task-template-builder.tsx`](../../apps/erify_studios/src/components/task-templates/builder/task-template-builder.tsx); Grid view renders rows = loops √ó columns = positional checkbox slots (Slot 1..N) with inline label + optional description per cell. Cards view stays canonical for structural changes (add/remove non-checkbox field, change field type, edit validation/options/conditional logic, drag-reorder). Both views edit the same `BuilderTemplateSchemaType` via the same `onChange`; no schema change. **Decision (locked)**: ship **Approach A only** ‚Äî positional grid + paste-TSV at the anchor cell (extends loops/columns as needed) + fill-column-down + duplicate-row. **Checkbox-only**: non-checkbox fields in a loop render read-only beneath the loop row with an "Edit in Cards view" link, so the grid stays a uniform cell editor. Approach B (key-aligned columns + per-cell apply-to-column / apply-where-equal-to-*X*) and Approach C (raw-JSON drawer) are documented in the design doc as deferred follow-ups ‚Äî Approach B is picked up only if producers report label-drift cases that A's fill-down doesn't catch; Approach C is not committed. Mobile = forced Cards view (same `useIsMobile()` pattern as PR 13). Shared fields locked from grid edits (lock icon mirrors Cards' "Shared" badge; paste lands only on non-shared cells, with a toast for the rejected ones). Full design at [`apps/erify_studios/docs/design/TASK_TEMPLATE_LOOP_GRID_VIEW_DESIGN.md`](../../apps/erify_studios/docs/design/TASK_TEMPLATE_LOOP_GRID_VIEW_DESIGN.md).

### PR 17 ¬∑ Creator-rate wire-type migration

**Brief** ‚Äî Surfaced during PR 5 review (Codex P2 on `updateStudioShowCreatorInputSchema`). `defaultRateInputSchema` and `defaultCommissionRateInputSchema` in `packages/api-types/src/studio-creators/schemas.ts` are both `z.coerce.number().‚Ä¶` (the canonical creator-rate wire format), so any decimal string sent from the FE is coerced through a JS `number` before the controller converts it back to a `Prisma.Decimal` string at the persistence boundary. For realistic rate values this is exact (everything is many orders of magnitude below `Number.MAX_SAFE_INTEGER`), but it's the last creator-side money path going through JS number and is inconsistent with the `Big`/string contract finalized in PR 1 + Finance Guardrail #2. Migrate both helper schemas to `z.string()` end-to-end across all three consumers (`createStudioCreatorRosterInputSchema`, `updateStudioCreatorRosterInputSchema`, `updateStudioShowCreatorInputSchema`), keep the `superRefine` cross-field invariants, update the controllers/services to parse decimals at the boundary (drop `toMoneyString` round-trip on the way out), and update the studios FE forms / payload builders (`buildShowCreatorAssignmentTermsPayload`, `buildUpdateStudioCreatorRosterPayload`) to send strings directly. Pair with PR 8 for the member-side equivalent ‚Äî same shape of change, same rationale. No DB migration required (creator rate columns are already `Decimal` in Prisma).

### PR 18 ¬∑ System creator rate wire-type migration

**Brief** ‚Äî Surfaced during PR 8 audit (2026-05-18). `packages/api-types/src/creators/schemas.ts` exposes the system-admin creator CRUD. The response schema (`creatorApiResponseSchema`) already serializes `default_rate` and `default_commission_rate` as `z.string().nullable()`, but the input schemas (`createCreatorInputSchema`, `updateCreatorInputSchema`) coerce them through `z.coerce.number()` ‚Äî so an admin POST/PATCH still round-trips money through a JS number even though the GET is string-safe. Migrate both input schemas to `z.string()` end-to-end, update the BE service/controller for `/admin/creators/*` (use `decimalToString` on the way out, parse decimals at the persistence boundary), and update any FE admin form that posts to these endpoints. No DB migration required (creator rate columns are already `Decimal` in Prisma). Independent of PR 17 (PR 17 covers the studio-scoped roster helpers); ship in any order.

### PR 14 ¬∑ Operator-recorded show actuals via CLOSURE task template

**Brief** ‚Äî Surfaced during PR 4 retrospective. PR 4 made show `actual_start_time` / `actual_end_time` editable on `/show-operations`, but that surface is for admin/manager audit. The on-the-ground source of truth for actuals is the operator running the show, who should record them as part of the post-production (CLOSURE) task. This PR moves operator capture into a CLOSURE task template assigned per show; admin/manager `/show-operations` remains as the audit/override surface (last-write-wins, with manager edits flagged in the audit log). **Design required before code** ‚Äî open questions: (1) CLOSURE task field schema for `actual_start` / `actual_end` with timezone handling and the validation rules from `show-actuals-dialog` (no inverted ranges, etc.); (2) role permissions on `Show.actual_*` writes (operator via task submission only, admin/manager via direct dialog); (3) reconciliation policy when both submit (last-write-wins vs. operator-canonical-with-manager-flag); (4) UX on `/show-operations` when an operator has already submitted (badge + read-only display until manager explicitly overrides); (5) backfill path for shows already closed under the old workflow. Design doc to land in `apps/erify_studios/docs/` before this PR opens.

## Goal

Build the L-side (cost) of P&L on existing studio entities and finish studio operational autonomy so studios stop depending on `/system/*` routes. Phase 4 produces **read-only reference figures** тАФ no money moves, no payment processing, no bank reconciliation, no dispute workflow.

- тЬЕ **PR 12.4.4 тАФ Operational facts summary and filters** ([#111](https://github.com/allenlin90/eridu-services/pull/111)): Show Run Review (`/show-run-review`) summarizes confirmed extracted facts for an operational day/range тАФ show-actuals completeness, late/missing creators with reasons, active platform violations, and incomplete phase checks тАФ across four URL-synced DataTable tabs (creators / violations / tasks / shows) with per-tab search and status filters. Operational-day window math moved fully to the frontend: the client computes 06:00тЖТ05:59 local bounds via shared `operational-day-range` utilities and serializes absolute ISO-8601 strings; the backend `GET /studios/:studioId/shows/run-review` endpoint stays timezone-agnostic, validates the explicit bounds, and caps the window at 31 days to bound the in-memory aggregation (FE surfaces the returned validation message). **Read-only**: it reports the state of already-extracted `Show` / `ShowCreator` / `ShowPlatform` / `ShowPlatformViolation` columns and never writes them; actuals are populated upstream by the extraction pipeline (12.0.5 / 12.1.1) on task approval (12.4.3). Show-actuals reporting is **start-based**: a show counts as started once `actualStartTime` is recorded (the primary lateness signal), and the card surfaces late-start count plus total missing duration (`actualStartTime тИТ planned startTime`); end-time recording is shown as secondary detail rather than gating completeness, since `show_actual_start_time` and `show_actual_end_time` are independent fact-key bindings.
  - **Follow-up тЖТ 12.4.6**: review surfaced that bulk approval silently extracts nothing when a completed task's frozen snapshot has no fact-key binding (e.g. generated before the binding was added) or empty bound fields, leaving actuals empty with no explanation. Surfacing "0 facts will be extracted" / flagging snapshot binding-drift at `/task-review` is folded into 12.4.6's stale-input resolution scope (deliverable added to [PRD ┬зC 12.4.6](../prd/task-fact-binding.md#section-c-review-surface-pr-124)).

Client-side serialization via the shared `@/lib/csv` + `@/lib/file-download` utils; no new endpoint. Per-tab pending/disabled state; no-op when the filtered total is 0.
- тЬЕ **Perf pass тАФ task-setup / task-review / show-run-review** ([#119](https://github.com/allenlin90/eridu-services/pull/119)): not a roadmap deliverable тАФ a cross-cutting optimization of the 12.4.x surfaces. `/task-review` tab badges move from client-side aggregation of a fetch-all dataset to a single `GET /tasks/review-stats` count endpoint, and the table tabs become server-filtered via `review_tab`; the stats query scopes by "due-date-in-range OR undated-with-show-in-range" so undated review tasks are still counted. Show Run Review splits its monolithic nested response into a lean summary plus four lazy paginated sub-resources (`run-review/{creators,violations,tasks,shows}`), with summary counts and list rows derived from shared helpers so they can't drift. Task lists drop the heavy JSONB `snapshot.schema` from the studio review list (lazy-loaded via the task detail endpoint) while the member `/me/tasks` execution path keeps it (`taskListIncludeWithSchema`). Pattern + guard rails documented in [`.agent/skills/api-performance-optimization/SKILL.md` ┬з8](../../.agent/skills/api-performance-optimization/SKILL.md).

### PR 15 ┬╖ Creator-rate wire-type migration

**Brief** тАФ Surfaced during PR 5 review. `defaultRateInputSchema` / `defaultCommissionRateInputSchema` in `packages/api-types/src/studio-creators/schemas.ts` are `z.coerce.number()`, so any decimal string from the FE is coerced through a JS number before the controller re-stringifies it.

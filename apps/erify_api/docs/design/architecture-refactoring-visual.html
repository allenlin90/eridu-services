02 ▸
How it differs

Reverse the primary axis: a capability owns the whole use case

Admin, studio, me, and MCP stay as transport & authorization. They stop dictating where domain logic lives. Everything that changes together moves behind one capability's public API.

Today · path from route to rule
Controller→ OrchestrationService→ ModelService→ Repository→ BaseRepository→ Prisma

…and where a repo is exported, callers skip the middle entirely:

Controller→ Repository (bypass)→ Prisma

6 hops, an inconsistent seam, and invariants that can be routed around.

Target · one deliberate public surface
Controller / MCP tool→ Capability API→ Use case · *Queries→ Persistence (tx-aware)

Persistence is private. A capability exports a small, intentional service/query surface — never a general repository. If a model service can't do an atomic op, the capability adds an owned persistence method instead of exporting the repo.

Fewer hops, a single owner, no bypass door.

The distinction that matters

This is strategic DDD — group by capability and shared language — with tactical DDD only where a real invariant exists. We are not wrapping every Prisma row in an aggregate, repository interface, mapper, and factory. Reference data (show types, statuses, platforms) stays plain CRUD.

03 ▸
The framework philosophy

Why NestJS isn't organized like Rails — and why that fits us

The reshape above isn't a style preference. It follows from what NestJS actually is, and from how it differs from the Rails / Active Record tradition of MVC.

In the Rails tradition, the model is the center of gravity. Active Record fuses three things onto one class — the table row, how it persists itself, and the business behavior around it. The advice is "fat models, skinny controllers," and the organizing axis is the database table: one model per table, and domain logic accretes on the model.

NestJS descends from a different lineage — Angular. Its building blocks are dependency injection, modules, and providers. The unit of encapsulation is not the class, it is the module: a boundary that declares providers, imports other modules, and exports a deliberate public surface. Business logic lives in injected services, and — with Prisma — the database row is an inert typed object with no behavior of its own (the Data Mapper pattern, the opposite of Active Record).

Why model-based encapsulation doesn't carry our domain

Active Record is excellent when a request maps cleanly to one row's lifecycle. Ours rarely do.

01Prisma is Data Mapper

There is no live model object to hang behavior on. Emulating Active Record over Prisma rebuilds the mapping layer Prisma deliberately removed — more code, more surprise.

02Invariants span many entities

publish() coordinates shows, tasks, snapshots, conflicts, and audit inside one 30-second transaction. That's an orchestrated use case with an explicit boundary — not a method on the Show row.

03Guarantees are cross-cutting

CLS transactions, optimistic locking, soft delete, audit, and UID boundaries live at service / repository seams. Active Record's before_save callbacks are a famously fragile home for them.

04Even Rails concedes

When logic outgrows the model, Rails reaches for service objects. Our capability use-case services formalize exactly that shape — from the start, not as an escape hatch.

An honest caveat

This isn't "NestJS forbids Rails." Nest can render MVC views, and TypeORM even offers an optional Active Record mode. The point is that Nest's grain (modules + providers), Prisma's Data Mapper, and our multi-runtime, cross-entity domain all pull the same way — toward capability-and-provider organization rather than model-and-row.

Why not just use Ruby on Rails, then?

Because we want Rails's locality — organize around the work, keep the path from request to rule short — but not Active Record's row-centric center of gravity, and we carry constraints Rails serves worse. The React frontends share one typed contract with the API through @eridu/api-types (Zod + Prisma as the source of truth); the whole monorepo is one language and one toolchain; the same domain must serve REST and an MCP server today — workers next — as separate runtimes on one provider graph; and our invariants are explicit multi-entity transactions, not per-row callbacks. Rails would fracture the stack and fight every one of those. The slice organization we want predates Rails and isn't Rails-owned — capability-first NestJS keeps the good part of Rails on a stack chosen for reasons Rails can't satisfy.

04 ▸
NestJS conventions

The idioms Nest rewards — and the reason behind each

These aren't arbitrary. Each convention buys a specific property: testability, encapsulation, or a short path from request to rule. The refactor leans further into them, it doesn't invent new ones.

1
Group by feature, not by type
Why Cohesion and locality. A change lands in one module instead of being smeared across all-controllers / all-services / all-repositories folders. This is Nest's own module guidance.
2
Export a deliberate public API
Why A module's exports array is its contract. Keeping persistence private means callers physically cannot route around an invariant.
3
Put logic in providers, inject by constructor
Why The container manages lifecycle; dependencies swap for mocks in tests; each provider keeps a single responsibility.
4
Keep controllers thin
Why Transport is a detail. The same service must serve a REST controller, an MCP tool, and a future worker without change — so business rules can't live in the controller.
5
Depend on injection, not new
Why Decoupling and testability. Wiring is declared once at the module boundary, not hardcoded into every call site.
6
Don't over-globalize
Why A provider that's global-by-default turns the dependency graph into implicit soup. Share only what is genuinely shared; keep the rest module-private.
7
Match data-access depth to real complexity
Why Nest does not mandate a repository per entity — that's a community habit. A seam should exist for genuine persistence complexity, not as ceremony. This is the one convention the guide sharpens.
Where we already stand

We follow most of this already — Zod three-tier contracts, CLS transaction propagation, the task-orchestration facade over cohesive sub-services, and mostly module-private providers. The guide sharpens the two idioms this codebase under-applies by default: organize by capability rather than by table, and let repository depth follow persistence complexity. Nothing here abandons Nest — it commits to it.

What "done" looks like

A worked example: one capability, end to end

Abstract until you see it on a real slice. Here is show-operations — the richest capability — as it would look finished. Same endpoints, same Prisma schema, same behavior as today; one owner instead of three trees.

# everything that changes together lives in one module
show-operations/
├─ show-operations.module.ts      # exports ONLY the two public providers ↓
├─ show-operations.service.ts     # public WRITE facade — thin, delegates
├─ show-queries.service.ts        # public READ api — DTO projections
├─ internal/
│  ├─ show-lifecycle.service.ts    # canonical status state machine (roadmap item 18)
│  ├─ show-assignment.service.ts   # platform / creator assignment sync
│  └─ show-performance.service.ts  # performance correction workflow
├─ policies/
│  └─ show-transition.policy.ts    # pure: is this edge allowed? role tier? reason?
├─ persistence/
│  └─ show.repository.ts           # PRIVATE, deep: optimistic-locked + bulk writes
├─ http/
│  ├─ admin-show.controller.ts     # same routes + guards as today
│  └─ studio-show.controller.ts
└─ schema/
   └─ show.schema.ts               # Zod contracts + payload types (unchanged)

The module exports exactly two things — the only surface other capabilities and runtimes may touch:

// write side — task-oriented methods, never CRUD verbs
class ShowOperationsService {
  cancelShow(p): Promise<Show>
  transitionStatus(p): Promise<Show>
  createShow(p) · updateShow(p) · replaceAssignments(p) · correctPerformance(p)
}
// read side — one scoped query each, DTO-projected
class ShowQueries {
  getShow(uid, scope) · listShows(filter) · getPublishImpacts(filter)
  getShowForMcp(uid, membership)   // single scoped read, no second round-trip
}

The same capability, two runtimes

REST · write · cancel a show
  1. PATCH /studios/:id/shows/:id/cancel hits StudioShowController.
  2. Guards run (Throttler → JwtAuth → StudioProtected); membership is validated and the UID is decoded to an internal id at the boundary.
  3. Zod parses the body; the controller calls showOperationsService.cancelShow() and does nothing else.
  4. The facade delegates to showLifecycleService.transition(), which owns the single @Transactional().
  5. In one transaction: load with deletedAt:null, run the pure transition policy, write the optimistic-locked status (version bump), write the Audit row.
  6. The result maps back to a snake_case response carrying the UID; the internal id never leaves.
Enforced right here transaction boundary · soft delete · optimistic lock · UID boundary · audit — all at one seam, not scattered.
MCP · read · fetch a show
  1. The get_show tool calls showQueries.getShowForMcp(uid, membership) — nothing else.
  2. That's one scoped query returning a DTO. No second relation read.
  3. The MCP runtime imports only *Queries providers + the auth policy — it never boots the write-heavy task and extraction modules.
  4. Same visibility and scope rule as REST, because both go through the one capability owner.
Why it matters the 24-module MCP boot closure collapses to a handful, and REST and MCP cannot drift apart.
This is the whole refactor, on one slice

The finished module is the ten responsibilities that today live in the 1,421-LOC studio-show-management.service.ts, redistributed into a facade + three internal services + queries + policy + repository. No endpoint, schema, or behavior changes. And the precedent already works: task-orchestration.service.ts is exactly this shape today. The refactor is "make each capability look like task-orchestration already does."

06 ▸
Why it's better

Locality, honest seams, and less to hold in your head

The wins are comprehension, test setup, and boundary safety — not raw latency. The guide is explicit: folder moves don't earn a performance claim.

→Find a rule by its domain term. A new engineer locates business logic from the route or capability name, without spelunking three trees.
→No silent bypass. Invariants live behind a public API; no exported repository to route around.
→Shallow CRUD sheds ceremony. Fewer modules, files, and mocks while the stable service API stays put.
→Repositories become evidence-based. A seam exists because persistence is genuinely complex — not because a template said so.
→Leaner runtimes. MCP boots only read-side capabilities; the 24-module closure shrinks materially.
→Real rollback coverage. High-risk workflows gain characterization tests that unit mocks can't provide.
07 ▸
Why not a CQRS bus (yet)

Move to level 2, stop there until the gates are met

CQRS is a spectrum, not a switch. The stated problem is too many files and layers — a command/handler/query/handler/event per operation makes that worse before it helps.

We are here
LV 0–1
Semantic separation
Task-oriented writes, named query methods, same service & DB.
Target
LV 2
Provider separation
Write use-case services + *Queries providers with DTO projections. Same DB.
Gated
LV 3
Nest CQRS infra
Command / Query buses, handlers, optional sagas. Runtime dispatch indirection.
LV 4
Independent read model
Denormalized store, eventual consistency, an outbox to maintain.

Adopt @nestjs/cqrs for one capability only when ≥2 gates hold: a command with multiple evolving consumers, a saga-shaped workflow, materially different read/write models, a reliable outbox, or defined handler ownership. None hold today.

08 ▸
The new persistence rule

Repositories become optional and evidence-based

This rule became canonical after the ShowStatus pilot passed and T12 reconciled every skill, agent, memory, and doc that had required "repository for all DB access."

Simple single-model CRUD, one caller, no reusable query policy
service → tx.<model> direct
Repeated complex filtering, projection, pagination, aggregation
private *Queries provider
Atomic conditional write, optimistic lock, raw SQL, advisory lock, bulk lifecycle
private repository / *Store
Multiple persistence implementations or a real external adapter
port + adapters

Every option still preserves the invariants that protect correctness: deletedAt: null on active reads, transaction visibility through TransactionHost.tx, scoped write predicates, UID-only contracts, and audit + optimistic-lock behavior. One adapter is a hypothetical seam; two adapters are a real seam — don't add an interface just to mock.

Setting the record straight

The earlier decisions this guide corrects

None of these were reckless — each was a reasonable default at the time. The guide reframes them because the codebase grew past the assumption. Tags mark whether it's a confirmed bug, doctrine that drifted from the code, a reframed convention, or a process improvement.

reframed convention
Was A repository for every model — "repositories hide all DB access."
Now Optional and evidence-based: simple CRUD uses tx.<model> directly; a repository exists only for genuine persistence complexity.
Most repositories just mirror Prisma. Accepted after the ShowStatus pilot; keep repositories when complexity earns the seam.
reframed axis
Was Organize by database table (one module per model) and by HTTP audience.
Now Organize by business capability; table and audience become attributes of a slice, not folders.
27 model modules and audience trees scatter one use case; 74/90 modules are ceremony.
confirmed bug
Was BaseRepository is a safe shared abstraction to build on.
Now Fix or retire it: its generic restore() can't restore a row, and inherited methods bind to the non-transactional client.
Dormant only because every real flow overrides it. A latent trap for the next author.
doctrine drift
Was Doctrine: "repositories are private module internals."
Now Reality: 6 are exported, so callers bypass model-service invariants. Close the leak — persistence stays private.
The written rule and the implementation had silently diverged.
reframed convention
Was UtilityService is injected everywhere so it's testable.
Now Pure deterministic functions (UID, time-overlap) don't need DI to be tested; fold them into shared/util.
48 modules import it purely for two pure functions.
refined guardrail
Was Many modules or a large file is automatically a smell to split.
Now Split by cohesion and co-change, not by LOC. A deep repository, a cohesive large file, or anemic reference data is not an anti-pattern.
Guards against over-correcting the cleanup into new churn.
process fix
Was Architecture metrics written as prose that goes stale on the next PR.
Now pnpm architecture:signals regenerates every figure and self-stamps the commit; wired into pr-review and repository-health.
The guide's own first draft had already drifted — this closes that loop.
process fix
Was Treat the reorg as a scheduled big-bang migration and this doc as a parallel backlog.
Now Trigger-gated phases executed through the codebase-hardening-program lifecycle — one PR per item, behavior-preserving.
Avoids a half-migrated tree and a second, competing process authority.

What makes it hard

The honest risks — and how the plan contains each

This repo is driven by AI agents that navigate via skills and hardcoded paths. A structural move has failure modes a human-only team wouldn't hit. Each is named with its mitigation.

coexistence

Two layouts at once

src/models/ and src/capabilities/ live side by side for a long window, since paths move only when actively refactored.

Contained by Trigger-gating: nothing moves speculatively. Each slice is a small, reviewable PR with before/after parity.
agent doctrine

Skills assume the old paths

At the source snapshot, pattern-authority skills and docs hardcoded repository/service/controller paths and the "repository for all DB access" rule; T12 reconciled them.

Contained by A per-phase knowledge-sync table + a matrix acceptance gate naming all 7 doctrine locations, enforced same-PR by pr-ready.
no safety net

Zero E2E coverage today

Service tests mock repositories, so they can't prove real CLS rollback, soft-delete visibility, or module-boot closure before files move.

Contained by Phase 0a: an isolated real-DB harness (never the dev volume) gates every behavior-bearing change. It blocks the reorg, not the trivial fixes.
latent bug

BaseRepository.restore() is broken

It adds deletedAt: null to the row it means to restore — it can't restore a soft-deleted row. Dormant only because every real flow overrides it.

Contained by A Phase 1 correctness item gated on the harness, closing the tracked residual and its lazy-delegate sibling in the same PR.
drift

Numbers rot as the code moves

A static snapshot in prose goes stale the moment a PR lands — exactly what happened to the first draft of this guide.

Contained by pnpm architecture:signals regenerates every figure and self-stamps the commit; wired into pr-review and repository-health.
resolved

Roadmap collision

Phases 4–5 target the same code as roadmap item 18 — the repo already closed #236–#240 for uncoordinated reach into it.

Resolved by Item 18 is the explicit activation trigger for Phase 4, cross-linked both ways between guide and roadmap.
Bottom line for the team

The direction is settled and no alternative beat it on review. What you can act on now is small and safe: stand up the Phase 0a harness and take the independent Phase 1 correctness fixes. The big reorg stays parked behind real triggers — you'll never be asked to move a folder "because the architecture says so," only because a piece of real work already needs that boundary.
